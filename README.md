# Production RAG Knowledge Assistant

基于 Hybrid Search + Cross-Encoder 重排序的生产级知识库问答系统。

## 项目概述

Production RAG Knowledge Assistant 是一个面向生产环境的知识库问答系统，核心目标是解决企业中知识检索与问答任务中的三大痛点：大语言模型对垂直领域知识的幻觉问题、传统向量检索在关键词匹配场景下的语义漂移、以及复杂查询所需的可追溯性与可控性不足。系统接收用户查询后，通过混合检索与多层重排序定位最相关的文档片段，结合上下文对话记忆生成带来源引用的结构化解^答，并提供可量化的评估与可视化监控能力。

这是我的独立学习项目，重点探索 RAG 从原型走向生产所需的关键工程实践。选择此方向是因为当前大量的 RAG 开源项目停留在向量检索+基础提示词的 Demo 层面，缺乏对排名质量、中期记忆、来源可追溯性等关键环节的深度工程化。本项目的价值在于将学术中验证有效的 Hybrid Search（如 Pinecone 的 BM25+Dense 实践）与 Cross-Encoder 重排序（如信息检索领域的 BERT-Based Re-ranking）应用于完整的问答 Pipeline，并通过结构化的评估闭环持续监控与迭代系统表现。

## 核心功能

- **Hybrid Search（向量 + BM25）**：系统同时执行稠密向量检索（基于语义近似）与稀疏 BM25 检索（基于关键词匹配），通过调整两种检索结果的权重进行融合。此设计解决了纯向量检索在处理专业术语、编号、人名等关键精确匹配时的召回不足问题。
- **Cross-Encoder 重排序**：在混合检索召回 Top-K 结果后，引入 Cross-Encoder 模型对结果进行高精度重排序。Cross-Encoder 能够直接在查询-文档对上进行联合建模，相比 Bi-Encoder（嵌入模型）具有更高的相关性判别准确度，显著改善最终送入 LLM 的上下文质量。
- **Metadata 过滤与来源高亮引用**：文档存入向量库时附带结构化 Metadata，检索阶段支持基于 Metadata 字段（如文档类型、标签、时间）的过滤，确保检索范围可控。生成答案时，系统自动定位引文来源并在原文中高亮对应段落，展示引用置信度。
- **对话记忆机制**：系统维护多轮对话历史的摘要与关键实体提取，在生成阶段将对话历史与新查询融合，而非简单地拼接原始对话文本。支持指定对话窗口长度的上下文管理策略。
- **评估与可视化**：内建基于 RAGAS 框架的标准评估，同时自定义 Chunk-level 与 Query-level 指标（如答案与文档块的重叠度），在 Streamlit 面板中直观展示评估结果、性能波动与历史趋势。
- **多格式文档解析与智能分块**：支持自动解析 PDF、Markdown、Word 等格式，应用基于文本语义边界的智能分块策略，在尊重段落/章节边界的前提下进行切片。
- **反馈闭环与持续优化**：收集用户对答案的 ThumbsUp/ThumbsDown 反馈，将负面案例自动加入优化候选队列。
- **可配置的 Pipeline 模块**：各模块（检索器、重排器、生成器等）均基于抽象接口实现，可根据业务需求灵活替换底层实现（如切换不同的嵌入模型或LLM）。

### 关键技术决策

#### 1. 为什么采用 Hybrid Search 而不是只用向量检索

**核心原因**：

向量检索（基于 Bi-Encoder 生成的稠密嵌入）擅长捕捉语义相关性，但对以下场景表现不佳：
- 包含特定 Id/编号、专有名词的精确查询。例如："文档编号 TS-2024-008 的内容是什么？"
- 短查询中关键词权重较高但语义不完整的场景。
- 需要精确短语匹配的长尾查询。

BM25 作为经典的词袋稀疏检索方法，在这些精确匹配场景下召回率更高。将两者结合（Hybrid Search），可以在保持深层语义理解能力的同时，弥补向量检索在精确性上的不足。

**实现方式**：
- 对同一查询分别执行稠密检索（Dense Retrieval）和稀疏检索（BM25 Retrieval）。
- 使用 RRF（Reciprocal Rank Fusion）对两种检索器的排序结果进行线性组合与重排序。
- RRF 公式：`score = Σ 1 / (k + rank)`，其中 k 为常数（通常取 60），rank 为各检索器中的排序位置。RRF 对原始分数进行归一化，避免了不同检索器分数分布差异带来的偏向。

**Trade-off**：
- **计算成本**：需要额外维护一套倒排索引（BM25），增加了索引阶段和检索阶段的计算开销。
- **索引复杂度**：需要同时维护两种索引结构，对数据更新同步提出了更高要求。
- **调参成本**：RRF 中 k 值的设定影响最终排名效果，需要结合具体语料进行评估调参。

**决策结论**：在通用企业知识库场景下，Hybrid Search 带来的整体检索质量提升明显大于额外开销，是生产环境中的推荐做法。对于纯语义匹配的闲聊类应用，纯向量检索可能已足够。

#### 2. 为什么加入 Cross-Encoder 重排序

**核心原因**：

Bi-Encoder（嵌入模型）在检索阶段通常将查询和文档都编码为低维向量，通过向量相似度进行排序。这种方法的两阶段性导致信息压缩：文档被编码为向量时语义可能丢失。Cross-Encoder 将查询和文档拼接后整体送入 Transformer 进行联合编码，可以在更细粒度上建模两者间的交互模式，其判别准确率显著高于 Bi-Encoder。

在标准的两阶段检索架构中：
- **第一阶段（Retrieval）**：追求召回率，使用快速的 Bi-Encoder 或 BM25 从海量文档中召回 Top-K 候选。
- **第二阶段（Re-ranking）**：追求精确率，使用计算成本更高的 Cross-Encoder 对候选集进行精细重排序。

**提升幅度**：

在典型的信息检索任务中，Cross-Encoder 相较于 Bi-Encoder 可以将 nDCG@10 提升 5% - 15%。在 RAG 这种端到端问答场景中，更精确的上下文排序可以直接降低下游 LLM 幻觉的可能性。

**Trade-off**：
- **显著的延迟增加**：Cross-Encoder 需要对每个候选对独立进行 Transformer 前向传播，对于 Top-100 候选集，延迟可能达到秒级，Bi-Encoder 通常在毫秒级。
- **计算成本**：需要为每个查询-文档对重新计算完整的 Attention（O(n²)），GPU 资源消耗较大。
- **超长文档截断**：Cross-Encoder 的输入长度受限（通常 512-1024 字符），对于超长文档需要截断或使用滑动窗口策略，可能丢失关键信息。

**决策结论**：将 Cross-Encoder 作为是否送入 LLM 的最后一道质量 gate 是必要的，但只作用于第一阶段召回的 Top-K（本项目取 Top-50），而非全量。用户可根据对延迟的容忍度动态调整 K 值。

#### 3. Metadata Filtering 的设计思路

**设计背景**：生产环境中，查询通常附加的业务约束需要被满足（如只搜索特定部门的文档、近一年内发布的文档）。如果只在检索后置过滤，不仅效率低下，还可能因为第一阶段召回不足而漏掉符合条件的文档。

**设计思路**：
- 文档入库时，将业务元数据（`department`、`doc_type`、`publish_date`、`category` 等）明确标注于向量库的 payload 中。
- 检索时，先应用 Metadata 预过滤器，将检索范围限制在符合用户鉴权与业务条件的子集内，再执行与 Hybrid Search 的交集计算。
- Chroma 和 Qdrant 均原生支持 `where` 子句或过滤表达式。

**Trade-off**：
- **索引设计复杂度**：需要预先规划 Metadata Schema，且 Schema 变更需要重建相关索引。
- **与 RRF 的耦合**：Metadata 过滤可能在第一阶段就剪掉了过多结果，影响 RRF 的融合效果。需在检索参数中调整召回数量，确保过滤后仍有足够条目参与重排序。

**决策结论**：将 Metadata 作为一级过滤条件（无数据税则索引命中率为0）而非排序因子，确保结果的合规性的同时，通过召回数量调参平衡业务结构与检索质量。

#### 4. 对话记忆方案设计

**核心问题**：如何管理多轮对话的上下文，既保证 LLM 理解用户意图，又避免超出模型上下文长度和引入过多无关信息。

**方案选型与** **为什么选择**：**

本项目采用 **动态滑动窗口摘要 + 稀疏实体向量记忆** 的混合方案，而非简单地拼接原始对话历史或粗暴截断。

1. **滑动窗口（Recent K Turns）**：
   - 保留最近的 K 轮（默认 K=3）完整对话原文。这些是最相关的上下文，直接使用不需要压缩。

2. **窗口外的历史通过 LLM 生成压缩摘要**：
   - 当历史对话超过 K 轮时，触发摘要生成。摘要中保留核心主题、用户最终意图、以及可能相关的关键事实（由 LLM 判断）。此操作利用了 LLM 的语义理解能力对历史进行有损压缩。

3. **关键实体独立记忆**：
   - 从历史对话中提取命名实体（人名、机构名、专业术语），维护一个独立的稀疏查询记忆池。当新查询涉及这些实体时，提升相关性权重。

**为什么不用其他方案**：
- **全量拼接原始历史**：直接导致提示词长度膨胀，超出模型上下文窗口；引入无关内容加重幻觉。
- **纯向量记忆**：虽然看似智能，但检索出的历史片段逻辑连贯性差，容易产生"断章取义"的偏差。
- **固定窗口截断**：丢失了早期对话中的关键约束（如"只查2023年后的报告"），导致后续轮次的回答偏离用户初始意图。

**Trade-off**：
- **摘要质量不稳定**：依赖 LLM 的摘要生成质量，如果 LLM 遗漏了早期轮次的关键约束，后续检索可能偏离目标。
- **引入了额外的 LLM 调用**：每满 3 轮触发一次摘要消耗，增加了成本和延迟。
- **状态管理复杂**：需要同时维护原始历史、压缩摘要、实体索引三个状态。

**决策结论**：在对话场景下，多轮内的语义连贯性（尤其是用户约束的保留）比成本更关键。此方案是当前兼顾质量与成本的最优平衡。

#### 5. 如何实现来源高亮和可追溯性

**核心目标**：让每个生成的事实都能被映射回原始知识库中的具体出处，且用户可以在UI中直观看到对应的高亮段落。

**实现路径**：
- **块级溯源**：知识库以 Chunk 为单位入库，每个 Chunk 附带唯一 doc_id 和 chunk_idx 以及原始文本内容。
- **引用生成**：LLM 生成答案时，被提示在关键事实后加入 `[source: doc_id, chunk_idx]` 格式的标记。
- **后端验证**：系统在返回前对 LLM 生成的引用进行验证，确保引用的 chunk 确实存在于检索结果中，杜绝 LLM 编造的"幻觉引用"。
- **前端高亮**：前端根据引用标记，在展示原文时自动将该段落背景高亮，并在悬停时显示相关片段。

**Trade-off**：
- **增加了提示词约束的复杂度**：要求 LLM 在生成流畅文本的同时严格遵循引用格式。
- **引用粒度**：Chunk 级别可能过粗（如果一个 Chunk 很长），后续可引入句子级精确定位。

**决策结论**：Chunk 级别的溯源是当前工程可行性与用户体验的最优折中。通过后端引用校验，可以在保证可追溯性的同时，最大限度地拦截 LLM 幻觉。

## 评估方法与当前结果

### 评估方案

1. **RAGAS 自动化评估**：Faithfulness、Answer Relevancy、Context Precision、Context Recall。
2. **LLM-as-Judge**：由更强的模型（如 GPT-4 或 Claude-3）作为评判者，对系统生成的答案进行综合评分。
3. **自定义指标**：
   - **Citation Accuracy（引用准确率）**：生成的引用中，有多少真实存在于检索结果中。
   - **Source Coverage（来源覆盖度）**：回答中是否覆盖了已知答案需要的所有关键来源。

### 评估维度

| 维度 | 定义 | 方式 |
|------|------|------|
| Faithfulness | 答案事实是否可被上下文支撑 | RAGAS |
| Answer Relevancy | 答案是否直接回应问题 | RAGAS + LLM Judge |
| Context Precision | 检索结果中相关 Chunk 的占比 | RAGAS |
| Context Recall | 生成正确答案所需信息是否被召回 | RAGAS |
| Citation Accuracy | 生成的引用是否有效 | 自定义规则 |

## 当前局限性与未来优化方向

1. **缺乏持久化会话管理**：当前对话状态仅保存在内存中，服务重启后丢失。计划集成 Redis 会话存储，并支持多用户隔离。
2. **Cross-Encoder 延迟较高**：在高并发的生产环境下，批次 Cross-Encoder 重排序存在显著延迟。计划引入量化模型（如 ONNX），并探索 Faiss + HNSW 的近似替代方案。
3. **评估数据集覆盖领域单一**：当前评估案例主要集中于技术文档，缺乏金融、医疗、法律等专业领域的评测数据。计划分领域构建并开源评估数据集。
4. **长文档处理与 Chunk 语义边界丢失**：简单的基于字符长度或固定重叠的分块方式可能破坏上下文。计划引入基于语义边界的智能分块策略（如使用 NLP 模型检测段落和主题切换）。
5. **缺乏自动化监控与告警**：当前缺少对检索质量、LLM 响应质量的实时监控。计划接入 Prometheus + Grafana，对关键指标进行实时告警。
6. **单节点部署限制**：当前所有组件部署在单个容器内，缺乏高可用性和水平扩展能力。计划拆分服务，引入微服务架构与负载均衡。
